## SimThing Doctrine And Invariants

- Everything is a **SimThing**; behavior is expressed as SimThings, properties, overlays, and admitted accumulator/resource-flow surfaces.
- After generation or import, the sole authoritative substrate is **`SimThingScenarioSpec`**: recursive `SimThing` root tree plus structural grid metadata, placements, links, and provenance.
- Studio projections, Bevy entities, GPU buffers, runtime reports, and editor config are **not** model authority.
- Model edits apply to scenario authority first; Studio rebuilds projection layers from authority.
- Terran Pirate is a **lower-layer golden fixture** only — not the canonical save-game tree shape.

## RF / Spatial-Tree Runtime Doctrine

```text
ScenarioSpec is the serializable authority.

ScenarioSpec
  -> Scenario root
     -> GameSession
        -> Owner SimThings
        -> GalaxyMap / WorldStateMap Location
           -> galactic gridcell Location SimThings
              -> interior local grid
                 -> star-system local gridcell Location SimThings
                    -> planet/starbase/orbital Location gridcell SimThings
                       -> cohorts / fleets / infrastructure / leaders as child SimThings
```

**Spatial invariants:**

- GameSession contains Owner SimThings and the root spatial Location.
- Owners are GameSession children, **not** spatial parents.
- Owner/RF identity is metadata/channel scope, not containment.
- Ownership changes update metadata/properties/columns, **never** spatial parentage.
- GalaxyMap / WorldStateMap is the root spatial Location.
- Every spatial gridcell is a Location SimThing with an interior local grid.
- Default interior grid is **1×1** unless expanded; star-system galactic gridcells use **10×10**.
- Inert gridcells still use receiver grids for falloff, emanation, and heatmap values.
- Planet/starbase/orbital bodies are local-grid Location cells inside star-system local grids.
- Cohorts, fleets, infrastructure, and leaders are child SimThings under spatial Location cells unless later promoted to spatial containers.
- Fleets move by spatial reparenting only when movement is explicitly unlocked.

**RF / Accumulator doctrine:**

- RF is generic flow accumulation over SimThings.
- **RF resolves locally at each parent Location gridcell node.**
- **Sibling surplus/deficit settles within the parent Location arena before net surplus/deficit bubbles upward.**
- Upward RF reduction is by owner/resource/scope.
- Owner/RF channels are metadata arenas/scopes, not spatial containment.
- AccumulatorOp / GPU rows are execution/proof/cache surfaces.
- ScenarioSpec remains the authority.
- CPU may perform oracle/reference/shadow/bookkeeping/report formatting.
- CPU must **not** become production simulation authority.

## Scenario Import / Save / Load Authority Model

- Save/load serializes the whole `SimThingScenarioSpec` via `serialize_scenario_authority` / `deserialize_scenario_authority`.
- Naked `root: SimThing` alone is insufficient without structural grid metadata, `map_container_id` binding, placements, links, and provenance.
- Loading must validate STEAD mapping, reserve existing `SimThingId`s, and reject duplicate or exhausted id space.
- Studio settings, camera state, and render settings may persist as editor metadata beside authority — they cannot reconstruct authority alone.

**STEAD mapping doctrine:**

- Scenario load must validate stable IDs and links.
- ID reservation must prevent collisions.
- Studio projection rows must derive from ScenarioSpec authority.
- Save/load must preserve stable Scenario identity, SimThing identity, link integrity, ownership metadata, RF metadata, and spatial tree shape.

### Rung 2 — LOADED-SCENARIO-STUDIO-SESSION-ENVELOPE-0

**Purpose:** Define the loaded Scenario runtime/session envelope used by Studio.

**Delivers:** Loaded ScenarioSpec authority digest; Studio projection readiness; STEAD/link/tree validation status; recursive RF readiness; save/export eligibility; runtime report sidecars; explicit statement that Studio config, Bevy entities, GPU buffers, and runtime reports are not Scenario authority.

**Exit:** Studio has a clean session envelope around ScenarioSpec authority and derived runtime/projection surfaces.

### Rung 3 — LOADED-SCENARIO-RECURSIVE-RF-RUNTIME-0

**Purpose:** Attach recursive Accumulator RF runtime to loaded ScenarioSpec spatial trees.

**Delivers:** Loaded scenarios compile recursive spatial-tree RF runtime reports; runtime walks actual Location tree; each parent Location gridcell node resolves local child/sibling surplus and deficit first; only net surplus/deficit bubbles upward; upward reduction is owner/resource/scope keyed; owner/RF channel scope remains metadata.

**Exit:** A loaded ScenarioSpec can produce correct recursive RF runtime reports from the spatial tree.

### Rung 4 — LOADED-SCENARIO-RUNTIME-REPORT-CHAIN-0

**Purpose:** Attach the existing recursive runtime report chain to loaded scenario sessions.

**Chain:**

```text
loaded ScenarioSpec
  -> recursive RF
  -> owner-silo/disburse-down
  -> local allocation
  -> local effects
  -> semantic projection
  -> semantic execution records
  -> delta previews
  -> runtime state rows
  -> runtime property view
```

**Rules:** Explicit runtime/report mode only; no automatic hidden mutation on load; default load remains non-simulating; runtime execution is user/agent-selected.

**Exit:** Studio-loaded scenarios can produce the full runtime report chain without savefile/persistent mutation.

### Rung 5 — SCENARIO-CANDIDATE-FROM-RUNTIME-0

**Purpose:** Generate a mutated candidate ScenarioSpec from runtime property-view rows.

**Delivers:** Clone loaded ScenarioSpec into candidate; apply runtime property-view rows to candidate only; original digest unchanged; candidate digest changes when mutation records exist; records preserve owner_ref, resource_key, scope, participant id, property id, before/runtime/after values.

**Rules:** No in-place mutation of loaded ScenarioSpec; no savefile write; no persistent history.

**Exit:** Runtime outputs produce deterministic candidate ScenarioSpec authority without touching the loaded original.

> **Note:** Headless candidate mutation proof already exists (SCENARIO-PROPERTY-MUTATION-AUTHORITY-BOUNDARY-0, #826). This rung wires it to **loaded** Studio session context.

### Rung 6 — SCENARIO-CANDIDATE-SAVE-REOPEN-0

**Purpose:** Save candidate ScenarioSpec and prove reopen roundtrip.

**Delivers:** Candidate canonical serialization; atomic temp-to-rename write; reopen saved candidate; validate STEAD IDs/links/tree shape; rebuild Studio projections; candidate digest before save equals reopened digest; runtime reports rebuilt from reopened ScenarioSpec, not persisted as authority.

**Exit:** A runtime-mutated candidate ScenarioSpec can be saved, reopened, validated, and projected in Studio.

### Rung 7 — STUDIO-SCENARIO-RUNTIME-SAVELOAD-UI-0

**Purpose:** Expose the completed scenario load/runtime/candidate-save/reopen path in Studio.

**Delivers:** Studio shows loaded ScenarioSpec digest; STEAD/link/tree validation status; recursive RF runtime readiness; candidate mutation availability; explicit Save Candidate and Reopen Candidate flow; existing Save Scenario / Load Scenario remain ScenarioSpec file controls; failed load/save preserves current session.

**Rules:** UI state is not authority; Bevy entities are not authority; runtime cache is not authority; Studio config remains presentation-only.

**Exit:** A user can load a scenario, validate it, run the explicit recursive RF/runtime path, save a candidate ScenarioSpec, and reopen it in Studio.
